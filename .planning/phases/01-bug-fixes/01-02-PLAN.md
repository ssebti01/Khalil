# Plan 01-02: Fix Movement Drift (Player Keeps Moving After Key Release)

## Goal

Players currently continue sliding after releasing a movement key. Fix the drift so the player stops within ~9 frames (150ms at 60fps) of releasing the key, and also stops immediately if the window loses focus.

## Context

**Root cause — epsilon-less damping.** In `Player.update()` the `else` branch (no key held) applies a 0.75× velocity multiplier each frame:

```javascript
const dampedVX = body.velocity.x * 0.75;
this.sprite.setVelocity(dampedVX, currentVY);
```

The series `v × 0.75^n` approaches zero asymptotically but never reaches it. After 20 frames the player still has ~0.03 px/frame of residual velocity — invisible in physics but enough to cause perceptible drift on the rendered sprite over time.

**Secondary cause — focus-loss drift.** When the browser tab loses focus (alt-tab, click away), `keyup` events are not delivered. The last held key remains `isDown = true` in Phaser's key state, so the player keeps receiving full `runSpeed` velocity every frame even with no physical key held. Velocity must be zeroed when the game loses focus.

## Must Haves

- [ ] Releasing the movement key stops the player visually within about 9 frames with no perceptible continued slide
- [ ] Alt-tabbing away and back does not cause the player to drift or run into a wall
- [ ] Normal movement (A/D or left/right arrow) feels identical to before — no jerkiness, no premature stop while the key is held

## Tasks

### Task 1: Add epsilon hard-stop to movement damping

**File:** `src/entities/Player.js`

**Action:** In `update()`, change the `else` branch so that once the damped velocity drops below 0.5 px/frame in magnitude it is clamped to exactly 0. This stops the player within ~9 frames after key release with no visual jerk.

**Details:**

Find the `else` block at lines 68-72:

```javascript
} else {
  // Let friction slow horizontal movement naturally — only damp, don't zero
  const dampedVX = body.velocity.x * 0.75;
  this.sprite.setVelocity(dampedVX, currentVY);
}
```

Replace with:

```javascript
} else {
  // Damp toward zero; hard-stop below epsilon to prevent infinite drift
  const dampedVX = body.velocity.x * 0.75;
  this.sprite.setVelocity(Math.abs(dampedVX) < 0.5 ? 0 : dampedVX, currentVY);
}
```

**Why 0.5 px/frame?** At `runSpeed = 5.5`, after releasing the key the sequence is 5.5 → 4.1 → 3.1 → ... → 0.5 (frame 9) → 0. The snap from 0.5 to 0 is about 8px/s — not visible to the eye. Going higher (e.g. 1.0) would feel like an abrupt stop; lower (e.g. 0.1) extends drift noticeably.

### Task 2: Zero velocity on window focus loss

**File:** `src/entities/Player.js`

**Action:** Register a `blur` listener on the game canvas in `_create()` so that when the player alt-tabs or clicks away, the player's velocity is immediately zeroed and Phaser's key states are cleared.

**Details:**

In `_create(x)`, after `this.sprite.setDepth(20)` (line 42), add:

```javascript
// Zero X velocity if window loses focus — prevents held-key drift on alt-tab
// Only X is zeroed so mid-air jump arcs are not interrupted by focus changes
this.scene.game.events.on('blur', () => {
  this.sprite.setVelocityX(0);
});
```
Note: This is additive to the primary fix (Task 1). Task 1 alone fixes the root cause (asymptotic drift). This task eliminates the secondary edge case of held-key drift on alt-tab. Task 1's correctness does not depend on Task 2.

`this.scene.game.events` emits `'blur'` when the browser window loses focus (Phaser built-in). This is the correct hook — `window.onblur` would require teardown boilerplate; `game.events` is cleaned up automatically with the scene.

No changes needed to the public API or constructor. Both players (p1 and p2) instantiate `Player` separately, so each registers its own listener and stops independently.
